Problem Description
ONNX Autopsy: Recovering Neural Architectures from Optimized Binary Graphs
Domain: Sequence-to-Sequence | Difficulty: Hard | Scoring: ↑ Higher is better

Overview
Every neural network you deploy goes through a one-way transformation: the ONNX graph optimizer restructures the model's computation graph for hardware efficiency, erasing the human-readable architecture in the process.

Conv2d + BatchNorm2d + ReLU collapses into a single opaque fused kernel — two layers vanish from the binary.
GELU becomes five arithmetic nodes, or zero, depending on the optimization path.
MultiheadAttention shatters into a tangle of MatMul, scale factors, and Softmax operations.
Dropout disappears entirely — removed as a training-only artifact.
Your challenge: given only the raw bytes of an ONNX computation graph after ORT Level-3 optimization, train a byte-level language model from scratch that can reconstruct the original PyTorch layer sequence — in exact forward-pass execution order.

This is a foundational, from-scratch model training challenge. Standard pretrained models are completely blind to ONNX bytecode. You cannot rely on zero-shot inference, pretrained encoders, or fine-tuning. You must build and train a custom byte-level architecture capable of inverting the ONNX graph optimizer's transformations.

The Pretraining Traps (Why Standard Models Fail)
The Tokenizer Massacre
Standard BPE tokenizers, trained on human language, have no concept of ONNX protobuf structure. The protobuf wire format uses varint-encoded field tags — every integer in the binary is encoded as (field_number << 3) | wire_type, a scheme utterly foreign to text tokenizers. BPE will slice through instruction boundaries, splitting semantically atomic byte sequences into meaningless fragments. You must build a custom byte-level tokenizer trained or designed specifically for this hex-encoded binary format.

The Graph Optimizer Scramble
ORT Level-3 optimization (ORT_ENABLE_ALL) transforms the ONNX graph in ways that make the original architecture unrecognizable:

Conv-BN fusion absorbs BatchNorm2d into Conv2d's bias — the BatchNorm layer disappears from the graph
Activation fusion merges ReLU, SiLU, and Mish into preceding convolution nodes
QKV projection fusion collapses three separate Linear layers (Query, Key, Value) into a single projection node
Dead node pruning removes Dropout entirely (training-only), yet the ground truth still includes it
The count of nodes in the optimized ONNX graph does NOT equal the count of layers in the original model. Your model must learn to reconstruct the hidden layer count and composition.

The Semantic Inversion Problem
Even if you write a custom binary parser and extract all op type strings from the optimized graph, you still cannot solve this challenge with rules alone. A single fused kernel could have come from 2, 3, or 4 original layers depending on the presence of BatchNorm, the specific activation, and whether Dropout was inserted between them. The inverse mapping requires learned inference from byte-level patterns — not lookup tables.

Quadratic Context Choking
Deep Transformer and ResNet architectures produce ONNX hex strings of 10,000–20,000 characters. Standard attention mechanisms will OOM at this context length. You will need FlashAttention-2, a linear-scaling architecture (Mamba, RWKV), or aggressive chunking strategies to handle the hardest examples.

Evaluation
Submissions are evaluated using Mean Normalized Edit Distance (NED) — the same metric used in the MIPS32 Autopsy challenge.

Because you are predicting a chronological sequence of layer executions, standard accuracy is insufficient. You receive partial credit for predicting the correct layer types with minor insertion, deletion, or transposition errors.

For a single row, NED is:

NED = 1.0 - LevenshteinDistance(predicted_list, true_list) / max(len(predicted_list), len(true_list))  
The final leaderboard score is the mean NED across all 2,000 test rows.

import json  
from Levenshtein import distance as lev_distance  # pip install python-Levenshtein  
  
def ned(pred_json: str, true_json: str) -> float:  
    try:  
        pred = json.loads(pred_json)  
        true = json.loads(true_json)  
    except Exception:  
        return 0.0  
    if not isinstance(pred, list) or not isinstance(true, list):  
        return 0.0  
    if not pred and not true:  
        return 1.0  
    dist = lev_distance(pred, true)  
    return 1.0 - dist / max(len(pred), len(true))

Rules
Hard Rules (violation = disqualification)
No pretrained weights. Your model must be initialized randomly and trained only on train.csv. HuggingFace checkpoints, PyTorch Hub models, timm, and any other pretrained weight source are prohibited.

No ONNX parsing libraries at inference time. The onnx and onnxruntime Python libraries are prohibited at inference time. Your model must ingest the raw hex string (or its binary equivalent) directly. A custom pure-Python binary parser is allowed as preprocessing but must be disclosed.

Custom tokenizer required. Pre-trained tokenizers (HuggingFace AutoTokenizer, tiktoken, SentencePiece, GPT-2 BPE, etc.) are prohibited. Your tokenizer must be designed or trained from scratch for this hex-encoded binary format.

No external data. You may not use any external dataset, corpus, or synthetic data source during training. Only train.csv (and val splits derived from it) are permitted.

Reproducibility. Your training script must set torch.manual_seed(42) and produce the same submission when re-run. Include the full training code in your submission.

What IS Allowed
Any model architecture (Transformer, Mamba, RWKV, CNN, SSM, hybrid)
FlashAttention-2, custom CUDA kernels, mixed precision training
Any optimizer, learning rate schedule, or regularization
Curriculum learning (e.g., training on easy examples first)
Data augmentation on the training set (e.g., permuting layer order for augmentation — but note this may hurt performance if overdone)
Reading ONNX documentation to understand the format (for tokenizer design)
