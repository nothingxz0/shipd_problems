Overview
Email archives routinely lose the original subject line through forwarding, threading, or quoting. Reconstructing which subject belongs to which body is a useful retrieval and triage task for inbox search, legal discovery, and customer support workflows.

This challenge provides a synthetic corpus of short business style emails spanning five common workflow categories: business requests, meeting invites, project updates, sales pitches, and support follow ups. Each email has a body written from a category specific template and a paired subject line written in one of three styles that vary in how tightly the subject references the body content. The corpus is entirely synthetic: all names, companies, projects, products, dates, and monetary figures are fictional. Training entity pools and test entity pools share no values, so no specific person or company name appears in both splits.

Training data contains about 3000 emails, each shown as a single body paired with its one true subject.

The test set is organized into 205 blocks. Each block contains four email bodies and four subject lines labeled A, B, C, and D. The four subjects in a block are exactly the original subjects of the four bodies in that block, shuffled. Your task is to recover the full body to subject mapping for every block: assign each body the letter of its original subject.

All four bodies in a block share the same category, and all four subjects in a block were generated with the same difficulty tier. Category alone is not a discriminator within a block, so the task cannot be solved by category matching.

Training rows do not ship with block structure. Any validation blocks you want to construct during development must be built from training data by grouping training emails by category and shuffling their true subjects within the group.

Evaluation
Submissions are scored using a weighted block level Hamming accuracy across three hidden difficulty tiers. For each block, count the fraction of body to subject assignments that match the ground truth (values in {0, 0.25, 0.5, 0.75, 1.0}) and then average across blocks within each tier. Dataset
The public dataset lives under dataset/public/.

train.csv contains training emails with columns:
id integer identifier
category string label naming the email category
body string email body text
subject string original subject line paired with the body
test.csv contains one row per test body, 820 rows across 205 blocks:
block_id integer block identifier from 1 to 205
body_index integer body position within the block, one of 0, 1, 2, or 3
body string email body text
test_subjects.csv contains one row per subject within a block, 820 rows total:
block_id integer block identifier that matches test.csv
subject_letter string label, one of A, B, C, or D
subject string subject line text
sample_submission.csv shows the submission CSV format with a baseline assignment for every (block_id, body_index) pair.
Submission
Produce a CSV file with exactly three columns:

block_id integer block identifier from test.csv
body_index integer body position from test.csv, one of 0, 1, 2, or 3
assigned_subject string letter, one of A, B, C, or D, naming the subject believed to match this body
A correctly formatted submission looks like:

block_id,body_index,assigned_subject  
1,0,C  
1,1,A  
1,2,D  
1,3,B  
2,0,B  
2,1,A  
2,2,D  
2,3,C  
The ideal submission assigns each block's four subject letters as a permutation of {A, B, C, D}, one per body. Submissions that repeat letters within a block still grade: each body is scored independently against the ground truth letter, so partial credit is possible even for non permutations.

Requirements:

Every (block_id, body_index) pair in test.csv must appear exactly once
No missing values are allowed
assigned_subject must be one of A, B, C, D; other values count as wrong
